## 1. サービス概要

営業担当者が AI 顧客エージェントと音声でヒアリング（ディスカバリー商談）のロールプレイを行うトレーニングサービス。

### 1.1 コンセプト

- **同一見込み顧客**に対して複数回の商談を重ね、課題を段階的に掘り下げる
- AI 顧客は回をまたいで一貫した人物像を維持し、良いヒアリングに応じて「気づき度」が進行する
- **真の課題**は全回終了までユーザーに非表示。最後に答え合わせする
- 各回終了後、先輩・上司が評価を記入。全回完了後に総評を受け取る

### 1.2 登場人物

| ロール | 説明 |
|--------|------|
| 営業ユーザー | プログラム作成・各回設定・音声ヒアリングを実施 |
| 評価者（先輩／上司） | 各回の評価および全回終了後の総評を記入（UUID URL でアクセス） |
| システム連携 | HULFT Square（通知・整形書き戻し）、Slack（評価依頼・フィードバック通知） |

### 1.3 用語

| 用語 | 意味 |
|------|------|
| プログラム | 1 見込み顧客との商談全体（複数セッションの親単位） |
| ヒアリングセッション | 制限時間内の 1 回の音声やり取り（最小単位） |
| 真の課題 | AI 顧客内部の本質課題。全回完了までユーザー非公開 |
| 気づき度 | 顧客が真の課題にどれだけ気づいているか（0–100、回ごとに更新） |
| ラポール度 | 営業との信頼関係（0–100、回ごとに更新） |

## 4. データモデル

### 4.1 ER 概要

```
Program 1──1 CustomerProfile（固定・生成後不変）
        1──1 CustomerState（回ごと更新）
        1──N HearingSession
        1──N OverallReview

HearingSession 1──N Evaluation
```

### 4.2 テーブル定義

| カラム | 型 | 説明 |
|--------|-----|------|
| id | UUID | PK |

## 5. 状態遷移

### 5.1 プログラム（`ProgramStatus`）

```
created
  → in_progress（最初のセッション作成時）
  → all_sessions_done（全回のヒアリング完了時）
  → overall_review_requested（総評依頼送信時）
  → closed（HULFT 総評 artifact 書き戻し時）
```

**真の課題の開示条件**（`should_reveal_true_challenge`）:

- `all_sessions_done` / `overall_review_requested` / `closed` のいずれかで API が `true_challenge` を返す

### 5.2 ヒアリングセッション（`SessionStatus`）

```
not_started
  → in_progress（/start）
  → evaluation_requested（/end 正常終了）
  → evaluated（先輩が評価投稿）

not_started / in_progress
  → abandoned（/abort：異常終了・やり直し）
```

- 進行中セッションが存在する間は新規セッション作成不可
- `abandoned` セッションは一覧から除外され、同一回数を再作成可能

## 6. 業務フロー

### 6.1 エンドツーエンド

1. ユーザーが **設定画面**で分野・回数・顧客特性を入力しプログラム作成
2. Bedrock が `CustomerProfile` + `CustomerState` を自動生成
3. **事前設定画面**でその回の目標・制限時間を入力しセッション作成
4. **ロープレ画面**で `start` → WebSocket 接続 → PTT で音声対話
5. 制限時間到達またはユーザーが終了 → `end` API
6. バックエンドが分析・録音保存・状態更新・HULFT/Slack 通知
7. 先輩が `/reviewer/evaluations/{token}` で評価記入
8. ユーザーが `/evaluations` で履歴閲覧（真の課題は非表示）
9. 全回完了後、総評依頼 → `/reviewer/overall-review/{token}` で総評
10. ユーザーが `/overall-review` で総評＋真の課題を閲覧

### 6.2 1 ターンの音声パイプライン（PTT）

```
[クライアント] ptt_start → PCM 16kHz チャンク送信 → ptt_end
       │
       ▼
[WebSocket] Transcribe ストリーミング STT
       │
       ▼
Bedrock ストリーミング応答生成（CustomerProfile + State + 目標 + 残り時間）
       │
       ▼
文単位 Polly TTS → バイナリ MP3 を WebSocket で送信
       │
       ▼
[クライアント] Web Audio / HTML Audio で再生
```

- セッション通しの PCM はメモリバッファに蓄積し、終了時に WAV 化して S3 保存
- 残り時間が `SESSION_TIME_WARNING_SEC`（既定 120 秒）以下で `ping` 受信時、`time_warning` を返す

### 6.3 セッション終了処理（`SessionFinalizeService`）

1. 会話ログから transcript 組み立て
2. 録音を `recordings/{program_id}/{session_id}.wav` として S3 保存
3. Bedrock 分析モデルで awareness / rapport / disclosed_info / title / summary 更新
4. `formatted_transcript` を暫定生成
5. ステータスを `evaluation_requested` に更新
6. HULFT `session_complete` イベント送信
7. Slack 評価依頼通知
8. 全回完了時: `overall_review_requested` + 総評 HULFT/Slack 通知

## 7. API 仕様

ベース URL: 開発時 `http://127.0.0.1:8000`、本番は ALB DNS。

### 7.1 ヘルスチェック

| Method | Path | 説明 |
|--------|------|------|
| GET | `/health` | `{"status":"ok"}` |

### 7.2 プログラム

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/programs` | プログラム作成（LLM ペルソナ生成） |
| GET | `/api/programs/{program_id}` | プログラム詳細取得 |
| POST | `/api/programs/{program_id}/upload-material` | 資料アップロード（pdf/txt/md、最大 10MB） |

**POST `/api/programs` リクエスト例:**

```json
{
  "field": "金融 / 信用金庫・地銀",
  "total_sessions": 3,
  "evaluator_ids": [],
  "user_id": "default-user",
  "personality_type": "慎重・課題にまだ気づいていない",
  "sub_field": "信用金庫・地銀",
  "it_knowledge_level": "平均的（一般的なPC操作やビジネスツールは問題なく使える）"
}
```

### 7.3 セッション

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/programs/{program_id}/sessions` | セッション作成 |
| GET | `/api/programs/{program_id}/sessions` | セッション一覧 |
| GET | `/api/sessions/{session_id}` | セッション詳細 |
| POST | `/api/sessions/{session_id}/start` | 開始（`in_progress`） |
| POST | `/api/sessions/{session_id}/end` | 正常終了・ファイナライズ |
| POST | `/api/sessions/{session_id}/abort` | 異常終了・破棄 |

### 7.4 評価（トークン認証・ログイン不要）

| Method | Path | 説明 |
|--------|------|------|
| GET | `/api/review/{token}` | 先輩用評価ページデータ |
| GET | `/api/review/{token}/recording` | 録音バイナリ（audio/wav 等） |
| POST | `/api/review/{token}/evaluations` | セッション評価投稿 |
| GET | `/api/review/overall/{token}` | 総評ページデータ |
| POST | `/api/review/overall/{token}/reviews` | 総評投稿 |

**評価投稿ボディ:**

```json
{
  "evaluator_id": "reviewer-001",
  "content": "ヒアリングの深さが良かったです。"
}
```

### 7.5 フィードバック

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/feedback` | ユーザーフィードバックを Slack へ送信 |

### 7.6 内部 API（HULFT 書き戻し）

| Method | Path | 認証 | 説明 |
|--------|------|------|------|
| POST | `/internal/evaluation-artifacts` | `X-API-Key` ヘッダ | 整形 transcript の DB 反映 |

**リクエスト例:**

```json
{
  "session_id": "uuid",
  "program_id": null,
  "formatted_transcript": "# ロープレ記録\n\n...",
  "artifact_type": "session"
}
```

`artifact_type: "overall"` + `program_id` の場合、プログラムを `closed` に遷移。

## 8. WebSocket 仕様

### 8.1 接続

```
WS /ws/sessions/{session_id}/hearing
```

- セッションが `in_progress` のときのみ接続可能
- 開発時: Vite が `/ws` をバックエンドへプロキシ

### 8.2 クライアント → サーバー

| type | 内容 |
|------|------|
| `ptt_start` | `media_format: "pcm_s16le"`, `sample_rate: 16000` |
| （バイナリ） | PCM 音声チャンク |
| `ptt_end` | ターン終了・STT/LLM/TTS 処理開始 |
| `ping` | 残り時間チェック（time_warning 用） |

### 8.3 サーバー → クライアント

| type | 内容 |
|------|------|
| `transcript_partial` | STT 途中結果 |
| `transcript` | 確定 transcript（`speaker: "user" \| "ai"`） |
| （バイナリ） | Polly MP3 チャンク |
| `turn_complete` | 1 ターン完了 |
| `time_warning` | `remaining_sec` 付き残り時間警告 |
| `error` | エラーメッセージ |
| `session_ended` | セッション終了（将来拡張用） |

## 10. 外部連携

### 10.1 HULFT Square

**バックエンド → HULFT（トリガー）**

| イベント | タイミング | ペイロード主要項目 |
|----------|------------|-------------------|
| `session_complete` | セッション終了 | session_id, review_url, field, goal, transcript, true_challenge |
| `overall_review_request` | 全回完了 | program_id, review_url |

認証: HULFT Login API → Refresh Token で REST API トークン取得。

**HULFT → バックエンド（書き戻し）**

- `POST /internal/evaluation-artifacts` + `X-API-Key`
- 同一 session_id への再 POST は上書き（冪等）

### 10.2 Slack

| 用途 | 環境変数 | 送信内容 |
|------|----------|----------|
| フィードバック | `SLACK_FEEDBACK_CHANNELS` | ユーザーメッセージ |
| 評価・総評依頼 | `SLACK_EVALUATION_CHANNELS` | 評価ページ URL |

`SLACK_BOT_TOKEN` を共用。チャンネル未設定時は Bot が参加しているチャンネルへ自動投稿（`conversations.list`）。

## 12. 開発・運用手順

### 12.1 パターン A: フロントのみ（推奨）

```powershell
cd frontend
copy .env.example .env.development
npm ci
npm run dev
```

→ http://127.0.0.1:5173  
バックエンドは共有 ECS（ALB）を利用。

### 12.2 パターン B: フロント + バックエンド完結

```powershell
# 1. PostgreSQL
cd backend
docker compose up -d

# 2. Python 環境
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt -r requirements-dev.txt
copy .env.example .env

# 3. マイグレーション
.\scripts\migrate.ps1

# 4. バックエンド起動
.\scripts\run_dev.ps1

# 5. 別ターミナルでフロント
cd frontend
npm ci
npm run dev
```

### 12.3 テスト

```powershell
# フロント
cd frontend && npm run ci

# バックエンド
cd backend
$env:APP_ENV="test"
$env:DATABASE_URL="sqlite+aiosqlite://"
$env:AWS_STUB_MODE="true"
$env:HULFT_STUB_MODE="true"
$env:INTERNAL_API_KEY="test-internal-key"
pytest tests/ -q
```

ルートの `./scripts/ci-check.sh` で両方実行可能。

### 12.4 CI/CD

| Workflow | トリガー | 内容 |
|----------|----------|------|
| `ci.yml` | push/PR to main | フロント lint + バックエンド pytest |
| `frontend-deploy.yml` | push to main（frontend 変更） | S3 sync + CloudFront invalidation |

バックエンド ECS デプロイは手動スクリプト（`backend/scripts/`）および `docs/aws-setup.md` 参照。

## 13. セキュリティ・制約事項

### 13.1 認証

| 対象 | 方式 |
|------|------|
| 営業ユーザー画面 | **なし**（演習スコープ） |
| 先輩評価ページ | UUID トークン URL（推測困難性） |
| HULFT 内部 API | `X-API-Key`（Secrets Manager 管理） |
| AWS | ECS タスクロール（アクセスキー埋め込みなし） |

### 13.2 既知の制約・注意点

| 項目 | 内容 |
|------|------|
| 録音バッファ | **ECS 単一タスク前提**のインメモリ保持（`sessions.py` の `_recording_buffers`）。マルチタスク化時は Redis 等が必要 |
| ユーザー ID | 固定 `default-user`。本番化時は Cognito 等が必要 |
| セッション中断 | 途中離脱は `abort` で破棄。WebSocket 切断だけでは自動 abort しない |
| ffmpeg | 非 PCM フォーマットの STT 変換にサーバー側 ffmpeg が必要 |
| 開発 DB | `APP_ENV=development` では Alembic なしでも `create_all` が走る |

### 13.3 機密情報管理

- API キー・Slack トークン・HULFT 認証情報は **`.env` / Secrets Manager のみ**
- リポジトリへのコミット禁止（`.gitignore` 済み）
